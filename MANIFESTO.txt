
Manifesto del modello

1. Il programmatore descrive il problema, non HTTP.

2. Il programma è algoritmico anche se gira sul web.

3. La concorrenza è responsabilità del runtime. 
    -> vero ?  Ad ogni chiamata deve rispondere un unico task -> lento.. Posso avere istruzioni intrinsecamente parallele eseguite lato server ? 

4. Lo space dati è parte del linguaggio. 
   -> soluzione amethist? Lo spazio dati viene rimpallato tra server e client e il server è stateless? mi piacerebbe ma comporta rischi e limiti.
   -> un motore di database può essere parte della sezione lato server ? se non altro per le prestazioni..
   -> rinominiamo le tabelle entity ? permettiamo un "Move" che fa la stessa cosa di lock,lock,read,write,unlock,unlock ?  
id_comune   -> obbligatorio, punta a tabella comuni
idz_comune  -> opzionale, punta a tabella comuni
e il compilatore/runtime può dedurre:
prefisso: id_
radice:   comune
tabella: comuni
nullable: no

oppure:
prefisso: idz_
radice:   comune
tabella: comuni
nullable: sì

A quel punto puoi scrivere:
table clienti = nome, id_comune;
table comuni = id, comune;

in(clienti, nome="Enrico")
{
    print comune.comune;
}   
----------------------- concetto di contesto
context libri
{
    table autori = nome, cognome;
    table libri  = titolo, id_autore;
}
context("libri");
inall("autori")
{
    ...
}
Errore:
Il campo id_autore fa riferimento alla tabella "autori"
che non è definita nel contesto "libri".

il context si porta dietro: 
_engine, _database, _user, _password che sono impostati a un default standard. 
Gli unici identificatori che iniziano con _ ammessi all'interno di un context sono _engine, _database, _user e _password. Qualunque altro identificatore con prefisso _ è un errore di compilazione.





5. La UI è un renderer, non il programma. 
   -> Separiamo programma e grafica. 
      Amethist compilava come -nw ma non si poteva usare la ask..  
      Basta costruire una struttura minimale e associare un CSS ?  
      Come lo integriamo nella fase di compilazione ? 
      Come ne permettiamo la modifica ? Stesso meccanismo di proposta->implementazione mantenendo l'originale?   

6. L'IA propone, il compilatore compila.
   -> costrutto di linguaggio con domanda, sistema, modello e in qualche modo collegamento API 

7. Il sorgente conserva l'intenzione.

8. Il backend è sostituibile (VM, C, WASM, cluster...)

9. Le primitive del linguaggio sono poche.

10. Ogni estensione deve poter essere spiegata in una pagina.

dataset persona =
{
    nome;
    telefono;

    indirizzi =
    {
        via;
        id_comune;
    };
};

In RAM è un dataset annidato.

Quando fai:

add(persona);

il runtime può creare/popolare implicitamente:

persone
  id_persona
  nome
  telefono

persone_indirizzi
  id_persona
  via
  id_comune

Quindi l’annidamento nel linguaggio diventa relazione nel database.

La regola potrebbe essere:

campo semplice     -> colonna
sotto-dataset      -> tabella figlia
id_padre implicito -> relazione automatica

Questo evita sia le liste ricorsive lumacose sia l’ORM pesante.

E resta molto Amethist: tu descrivi la struttura naturale del dato, il sistema decide come normalizzarla.

Punto chiave:

Un dataset annidato è una vista logica.
La persistenza lo traduce automaticamente in tabelle e relazioni.

prolog parentela
{
    padre(mario,luigi).
    padre(luigi,andrea).

    nonno(A,C) :-
        padre(A,B),
        padre(B,C).
}
goal(parentela, nonno(X,andrea))
{
    print X;
}
goal(scacchi, mossa(M))
using minimax(depth=4, score=valuta_posizione)
{
    print M;
}
goal(percorso, arriva(A,B))
using heuristic(costo_distanza)
{
    print strada;
}
goal(problema, soluzione(X))
using minmaxbounded(max_nodes=10000, timeout=2)
{
    print X;
}
