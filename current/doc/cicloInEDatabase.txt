IN() stateless su database
==========================

Problema
--------

  context aaa(nome,cognome);
  inall (aaa)
  {
    ask nome;
    delete(aaa,persona=nome);
  };

ASK interrompe il flusso e la VM non puo' conservare un cursore o una
connessione DB aperta. Inoltre piu' IN() possono essere annidate.

Soluzione
---------

QLIST decide una volta l'insieme e restituisce una lista serializzata di id.
La lista e la posizione corrente restano nel frame dell'IN() sullo stack.
QBYID rilegge per chiave il record corrente e valorizza direttamente le
variabili del context.

Il filtro serve soltanto a QLIST. QBYID non lo riceve e non lo riapplica:
l'appartenenza all'insieme e' gia' rappresentata dalla lista salvata.

Non usare mai la posizione come OFFSET SQL. L'ordinamento di QLIST e'
deterministico: se context._orderby e' vuoto viene usato id; altrimenti id
viene aggiunto in coda all'ordinamento richiesto.

Interfacce VM
-------------

QLIST
  POP(filter), POP(context), PUSH(context).
  A riceve la lista serializzata degli id.

QBYID
  POP(id), POP(context).
  I campi letti vengono copiati nelle variabili context.*.
  A vale "1" se il record esiste, altrimenti stringa vuota.

GETPATH
  POP(numero_livelli), POP delle chiavi, POP(nome_variabile).
  Nel ciclo IN estrae dalla lista l'id alla posizione corrente.

DECSP n e INCSP n
  Permettono al codice generato di muoversi dentro il frame dell'IN() senza
  distruggere i frame degli eventuali cicli esterni.

Schema concettuale
------------------

  lista = QLIST(context, filtro)
  posizione = 0

  loop:
    id = lista[posizione]
    se id e' vuoto: vai a fine

    trovato = QBYID(context, id)
    se trovato:
      esegui corpo IN

    posizione++
    torna a loop

  fine:
    rimuovi soltanto il frame appartenente a questa IN

Se un record viene cancellato dopo QLIST, QBYID restituisce stringa vuota e il
ciclo prosegue con l'id successivo. Ogni IN() annidata mantiene sullo stack la
propria copia di context, lista e posizione.
