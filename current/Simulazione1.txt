La ask, come al solito, genera un form che si porta dietro lo stato.

Il task può essere lanciato in cron o a intervalli. Il task viene lanciato lato server con una chiamata all'HTTP che lancia il programma e un entry point per il task.
La ask ferma il programma ma non il browser, la procedura che lancia i taget può essere un javascript con setTimout..   
Il target nonè altro che un lancio del programma con il contesto di ask e un entry point che fa eseguire il codice del task
Il target risponde con una pagina html che aggiorna il div. 
Il task ha uno stato running/stopped etc.. unico a sistema.
Il target ha lo stesso stato ma riferito al singolo ciclo di esecuzione. Intendo dire la sessione che un utente vede sul suo browser. Quindi il ciclo di esecuzione deve essere riconoscibile con un ID oltre che protetto con una firma. Penso che html preveda gia una specie di id di browser e pagina potremmo usarla. Vorrei evitare le sessioni perchè sono un forte vincolo per il parallelismo.


Task
-----
- lato server
- nessuna interfaccia
- stato globale
- schedulabile
- eseguibile manualmente
- unico o multiplo
- entry point del runtime

Target
-------
- lato client
- associato a un renderer
- aggiornato via AJAX
- stato per browser/ciclo
- può essere aggiornato periodicamente
- può essere aggiornato da eventi
